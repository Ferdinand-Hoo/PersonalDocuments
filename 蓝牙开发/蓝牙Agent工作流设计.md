# 0 引言：为什么需要蓝牙Agent
随着车载信息娱乐系统（IVI, In-Vehicle Infotainment）的不断发展，蓝牙已经成为车机系统中最重要的通信模块之一。蓝牙不仅承担着传统的电话通话、音频播放和设备连接等功能，还在车机互联生态中发挥着关键作用。例如 Apple CarPlay、华为 HiCar 以及 Carlink 等手机互联方案，在设备发现、连接建立以及控制通道中通常都会依赖蓝牙作为基础通信链路。因此，蓝牙模块的稳定性直接影响车载系统的用户体验。

![[蓝牙Agent引言配图.png]]

然而，在实际开发和调试过程中，蓝牙问题的定位往往依赖大量日志分析，例如 HCI 日志、蓝牙协议栈日志以及应用层日志。这些日志数据量巨大、结构复杂，其中既包含底层二进制数据，也包含多种协议交互信息。工程师在排查问题时往往需要逐行分析日志，手动还原连接流程，这不仅耗时较长，也对工程师的协议理解能力提出了较高要求。

随着 AI Agent 技术的发展，可以通过自动化工具对蓝牙日志进行解析和分析，从而辅助工程师进行故障定位。通过将原始日志进行结构化处理，并结合 AI 模型对协议行为进行推理，可以显著提升蓝牙问题分析的效率。本文提出一种蓝牙 Agent 工作流设计，通过日志采集、协议解析、连接时序构建以及 AI 分析等步骤，实现蓝牙问题的自动化分析。

# 1 蓝牙Agent整体架构设计
在蓝牙日志分析场景中，本文提出一种基于 AI Agent 的自动化分析架构。该架构以 Cursor 作为 Agent 运行环境，通过 MCP（Model Context Protocol）扩展工具能力，并结合日志解析模块与领域 Skill，实现蓝牙问题的自动化分析。

为了实现蓝牙日志的自动化分析与问题诊断，本文将系统划分为三个核心层级：日志采集层、日志解析层以及 SKILL 层。系统通过自动采集日志、解析协议数据并调用不同诊断 SKILL，实现从“日志获取”到“问题分析”的完整自动化流程。

# 2 日志采集层设计
在蓝牙问题分析系统中，日志采集是整个诊断流程的入口环节。只有获取到完整且连续的 HCI 日志，后续的协议解析以及问题诊断才能顺利进行。因此，日志采集层不仅需要具备基本的日志获取能力，还需要具备稳定性、自动化能力以及与 AI 诊断系统的可集成性。

在传统蓝牙问题调试过程中，开发人员通常需要手动登录车机系统，通过终端执行日志抓取命令获取蓝牙 HCI 数据，或者将日志文件导出到 U 盘后再拷贝到本地环境进行分析。这种方式虽然实现简单，但存在明显问题：
- 严重依赖人工操作，调试效率低
- 日志采集过程不可复用，难以标准化
- 无法与自动化分析系统直接集成
- 在远程调试或批量问题分析场景下难以使用
因此，本文在系统架构中设计了独立的日志采集层，用于实现蓝牙日志的自动化获取与传输，使 AI 系统能够直接触发日志采集流程，从而减少人工干预并提高调试效率。

## 2.1 设计目标与设计原则
日志采集层的设计目标并不仅仅是“获取日志文件”，而是为后续的自动化诊断提供稳定的数据来源。因此，在设计时需要满足以下要求：
- 支持自动触发日志采集流程
- 支持远程车机日志获取（无需人工登录设备）
- 支持标准化输出（统一的 BTSnoop 日志格式）
- 支持与日志解析层直接对接
- 支持后续扩展（例如自动压缩、自动上传、自动解析）

# 3 日志解析层设计
在日志采集层获取到蓝牙 HCI 日志之后，系统仍然无法直接进行问题诊断，因为原始的 BTSnoop 日志本质上是二进制数据，既不具备可读性，也无法直接用于自动化分析。因此，需要在系统架构中设计独立的**日志解析层**，用于将原始日志数据转换为结构化的协议记录，从而为后续的蓝牙问题诊断以及 AI 自动分析提供基础数据支持。

日志解析层的核心目标并不是简单地解析数据，而是将“原始日志”转换为“可理解的通信过程”。因此，该层不仅需要完成协议解析，还需要对解析结果进行结构化封装，使上层模块可以直接基于解析结果进行统计分析或问题诊断。

## 3.1 设计目标
日志解析层的设计目标主要包括以下几个方面：
- 将原始 BTSnoop 二进制日志转换为结构化数据
- 提供统一的数据结构（HCIRecord）供上层模块使用
- 支持蓝牙协议的分层解析
- 提供可扩展的协议解析框架
- 为 AI 自动诊断提供高质量输入数据

## 3.3 分层解析架构设计
日志解析层采用自上而下的协议分层解析方式，整体可以划分为以下三个层级：
### 3.3.1 文件层
文件层负责解析 BTSnoop 文件格式，并按照记录结构拆分数据，生成带有时间戳与原始包体的基础记录（Raw Record）。这一层的主要作用是将二进制日志文件转换为可处理的记录数据。
### 3.3.2 HCI层
在文件层得到 Raw Record 之后，系统根据数据包类型对数据进行进一步解析。HCI 层的核心职责是识别数据类型，并将数据分发到对应的解析模块。

![[HCI层数据分发架构.png]]
### 3.3.3 L2CAP层
对于 ACL 数据，系统会进一步解析 L2CAP 头部，并根据 Channel ID（CID）将数据分发到不同的协议解析模块。例如：

- 0x0001：L2CAP 信令通道（连接请求、配置请求等）
- 0x0004：ATT（Attribute Protocol）
- 其它 CID：预留用于扩展（SDP、RFCOMM、A2DP 等）

这种设计方式使得系统可以在不修改上层逻辑的情况下，持续新增新的协议解析能力。

# 4 SKILL层架构设计

## 4.1 设计目标与设计原则
在前文中，系统已经完成了日志采集和日志解析等基础能力，但这些能力本质上仍然属于数据处理层，无法直接完成蓝牙问题的自动诊断。因此，需要在系统架构中引入 SKILL 层，用于将蓝牙协议分析经验进行结构化抽象，使 Agent 具备稳定、可复用的自动诊断能力。

本章节的设计目标包括：
- 支持传统蓝牙问题的自动分析（配对失败、连接断开、音频异常等）
- 支持车机互联场景下的蓝牙问题分析（CarPlay、HiCar、CarLink 等）
- 构建可扩展的诊断能力体系
- 支持后续持续新增 SKILL

## 4.2 SKILL的能力模型设计
SKILL 并不是简单的脚本或 Prompt，而是一个完整的诊断能力单元。每一个 SKILL 都需要具备明确的输入、分析逻辑以及结构化输出。
![[SKILL能力模型设计.png]]
## 4.3 SKILL的分层与分类设计
为了保证系统可以长期扩展，需要在设计阶段对 SKILL 进行分层。
### 4.3.1 基础蓝牙问题分析类SKILL
这一类 SKILL 主要用于分析传统蓝牙问题，例如：
- 配对失败
- 连接失败
- 自动重连失败
- 蓝牙连接异常断开
### 4.3.2 车机互联场景类SKILL
在车载系统中，CarPlay、HiCar、CarLink 等手机互联功能通常以蓝牙作为入口，一旦蓝牙阶段出现异常，往往会表现为互联失败。因此，需要单独设计互联类 SKILL。
![[车机互联场景类SKILL.png]]
### 4.3.3 音频与控制类SKILL
这一类 SKILL 主要用于分析音频相关问题，例如：
- HFP 通话没有声音
- A2DP 音乐无法播放
- 绝对音量异常
- AVRCP 控制失败
### 4.3.4 兼容性与设备行为分析类SKILL
在实际项目中，很多问题并不是协议错误，而是设备行为差异导致的兼容性问题。因此，需要设计专门的兼容性分析 SKILL。

## 4.4 SKILL 的调用机制设计
当用户提出问题时，系统不会直接调用工具，而是先识别问题类型，并自动选择合适的 SKILL 进行分析。
